## 1. Introduction

### 1.1 Purpose

This document provides a comprehensive description of the User Interface (UI) wireframe designs for the Intelligent Energy Management System (EMS). It defines the structure, layout, interaction patterns, and design rationale for each screen in the application.

The document serves as a shared reference between UI/UX designers, frontend and backend developers, system architects, quality assurance engineers, and project stakeholders. It bridges the gap between the Software Requirements Specification (SRS) and the final visual design by translating functional requirements into concrete screen structures and interface behaviours.

### 1.2 Scope

This documentation covers wireframes for all primary screens of the EMS web application as derived from the SRS v1.0. It includes:

- All eight primary screens of the application.
- Annotations for UI components, interactions, and business rules.
- User flow descriptions for the most critical journeys.
- Responsive design guidance for desktop and mobile views.
- Accessibility and usability considerations.

This document does not cover detailed visual design specifications such as typography scales, brand colour palettes, or icon assets. Those are addressed in a separate UI Style Guide document.

## 2. Project Overview

### 2.1 Background

The Intelligent EMS is a building-level and multi-site energy monitoring and management platform designed for commercial buildings, hotels, and supermarkets in Kenya. It collects real-time electrical data from smart meters and IoT sensors, processes it into operational intelligence, and presents it through dashboards, reports, and automated or operator-approved control actions.

The Kenyan operating context introduces specific requirements that influence UI design decisions, including variable grid quality, mixed connectivity environments, cost sensitivity, and compliance with the Kenyan Data Protection Act, 2019.

### 2.2 Design Objectives

The UI is designed to achieve the following outcomes:

1. **Clarity** — Present complex energy data in a form that facility managers can interpret and act on without specialist technical training.
2. **Efficiency** — Allow common operational tasks such as reviewing alerts, drilling into device data, and generating reports to be completed in the fewest possible steps.
3. **Safety** — Ensure that control actions require deliberate user confirmation and that safety-critical assets cannot be inadvertently commanded.
4. **Trust** — Reinforce data accuracy through clear timestamps, data quality indicators, and audit trails.
5. **Scalability of Experience** — Support both single-building operators and multi-site portfolio managers through the same interface structure.
6. **Accessibility** — Meet basic accessibility standards so that the platform is usable by operators with varying levels of digital literacy.

## 3. Design Approach and Principles

### 3.1 Wireframe Fidelity

The wireframes in this documentation are **mid-fidelity**. They use:

- Structural layout boxes with real labels, not placeholder rectangles.
- Realistic data values to communicate data density and formatting.
- Greyscale colour coding to indicate state differences (normal, warning, critical).
- Annotations that describe behaviour, business rules, and edge cases.

Mid-fidelity wireframes were chosen to accelerate stakeholder review and developer alignment without committing to full visual design too early.

### 3.2 Core Design Principles

| Principle             | Application in EMS                                                   |
|-----------------------|----------------------------------------------------------------------|
| Hierarchy First       | The most critical KPIs are at the top of every operational screen    |
| Progressive Disclosure| Summary at portfolio level; detail available on drill-down           |
| Consistent Patterns   | Same navigation, card, table, and form patterns used across screens  |
| State Visibility      | Every device, alert, and system element shows its current state clearly |
| Fail-Safe Interaction | Destructive and control actions always require a confirmation step   |
| Role-Appropriate Views| Screens and data are filtered and presented by user role             |

### 3.3 Grid and Layout System

- **Desktop layout:** 12-column grid, 1440px wide canvas, 24px gutters.
- **Sidebar navigation:** Fixed 240px left sidebar for primary navigation.
- **Content area:** Remaining width, divided into sections using card containers.
- **Mobile layout:** Single-column stacked layout, collapsible hamburger navigation.

## 4. User Personas and Roles

The EMS UI serves four distinct user roles, each with different information needs and interaction patterns. Every screen described in this document specifies which roles can access it.

### Role 1 — Facility Manager

- **Goal:** Monitor building energy consumption in real time, review and respond to alerts, generate operational reports, and approve or execute control actions.
- **Usage Pattern:** Daily, high-frequency interactions across dashboards and alerts.
- **Key Screens:** Site Dashboard, Alerts Panel, Reports, Control Panel.
- **Access Level:** Full operational access to assigned buildings.

### Role 2 — Executive Viewer (Building Owner / Management)

- **Goal:** Review energy savings, cost performance, and compliance evidence at a summary level.
- **Usage Pattern:** Weekly or monthly, low-frequency interactions focused on reports and KPI summaries.
- **Key Screens:** Portfolio Dashboard, Reports.
- **Access Level:** Read-only access to summary data across authorized buildings.

### Role 3 — Technician / Engineer

- **Goal:** Onboard and configure devices, troubleshoot communication faults, verify commissioning, and configure control policies.
- **Usage Pattern:** Periodic or event-driven, focused on device and configuration screens.
- **Key Screens:** Device Management, Control Panel (configuration), Alerts Panel.
- **Access Level:** Device and configuration access; read access to operational data.

### Role 4 — System Administrator

- **Goal:** Manage user accounts, assign roles and buildings, configure system-wide settings, and review audit logs.
- **Usage Pattern:** Infrequent, administrative interactions.
- **Key Screens:** User Management, System Settings.
- **Access Level:** Full administrative access; no operational control of building loads.

#### 7.1.2 Component Descriptions

| Component          | Description                                                      |
|--------------------|------------------------------------------------------------------|
| Logo + App Name    | Branding element. Centres the user on the product context.       |
| Email Field        | Standard text input. Required field. Validated on submit.        |

#### 7.1.3 Interaction and Behaviour

- On failed login, display an inline error below the form: *"Invalid email or password. Please try again."*
- After three consecutive failed attempts, display a CAPTCHA or a temporary lockout message.
- On successful login, redirect the user to their role-default screen (Portfolio Dashboard for Admins and Executives; Site Dashboard for Facility Managers and Technicians).
- Authenticated sessions automatically terminate after 30 minutes of inactivity (NFR-10).
- All login and logout events are recorded in the audit log (NFR-11).
- All communication is over HTTPS/TLS (NFR-7).

#### 7.1.4 Edge Cases and Error States

| Scenario                         | UI Behaviour                                        |
|----------------------------------|-----------------------------------------------------|
| Empty field on submit            | Inline validation: "This field is required."        |
| Invalid email format             | Inline validation: "Enter a valid email address."   |
| Incorrect credentials            | Form-level error banner below inputs.               |
| Account deactivated              | Error: "Your account is inactive. Contact your administrator." |
| SSO provider unavailable         | Error: "Single sign-on is currently unavailable. Try again later." |

#### 7.4.4 Alert Lifecycle States

```
Open → Acknowledged → Assigned → Resolved → Closed
```

- Menu items not accessible to the current user role are hidden (not just greyed out) to prevent confusion and enforce RBAC (FR-41).
- The sidebar collapses to an icon-only mode on narrow viewports.

### 8.2 Breadcrumb Navigation

Every screen below the top level shows a breadcrumb trail:

```
Portfolio > Nairobi HQ > HVAC Zone A
```

Breadcrumb elements are clickable links back to the parent screen.

### 8.3 Confirmation Dialogs

All destructive, irreversible, or safety-relevant actions require a confirmation modal before execution (UI-5). This applies to:

- Control commands (ON / OFF / Setpoint).
- Retiring a device.
- Deactivating a user.
- Deleting an automation rule.

### 8.4 Export Behaviour

All screens with tabular data or charts provide export options:

- **CSV** — Raw data for all visible rows.
- **PDF** — Formatted report layout including chart screenshots.

## 10. Responsive Design Guidelines

### 10.1 Breakpoints

| Breakpoint  | Width           | Layout Behaviour                                 |
|-------------|-----------------|--------------------------------------------------|
| Desktop     | ≥ 1280px        | Full sidebar visible. Multi-column card layout.  |
| Tablet      | 768px – 1279px  | Sidebar collapses to icons. Cards reflow to 2-col.|
| Mobile      | < 768px         | Sidebar becomes hamburger menu. Single column layout. KPI cards stack vertically. |

### 10.2 Mobile-Specific Adjustments

- Charts render in a horizontal scrollable container on mobile to preserve readability.
- Data tables on mobile show only the 3 most critical columns; remaining columns accessible via row expansion.
- Control Panel is accessible on mobile but displays a stronger warning banner before enabling interaction.
- All touch targets are a minimum of 44×44px to meet usability standards.

### 10.3 Minimum Supported Resolution

The interface is fully functional at **1366×768px** as required by NFR-19.

## 11. Accessibility Considerations

| Consideration            | Implementation Guidance                                            |
|--------------------------|--------------------------------------------------------------------|
| Colour Contrast          | All text meets WCAG 2.1 AA minimum contrast ratio of 4.5:1.        |
| State Not Colour Alone   | States (Normal / Warning / Critical) use colour AND icon/label to avoid reliance on colour alone. |
| Keyboard Navigation      | All interactive elements are reachable via Tab key. Focus states are visible. |
| Screen Reader Support    | All form inputs, buttons, and status elements include ARIA labels. |
| Error Messages           | Form errors are described in text, not just red borders.           |
| Minimum Touch Target     | 44×44px minimum for all interactive elements on touch devices.     |
| Time-Limited Sessions    | The session timeout warning is displayed 2 minutes before automatic logout so the user can act. |

## 12. Design Rationale

| Decision                                    | Rationale                                                                          |
|---------------------------------------------|------------------------------------------------------------------------------------|
| Fixed left sidebar navigation               | Provides persistent access to all sections on operational screens where context switching is frequent. |
| KPI cards at the top of every screen        | Facility managers need the most critical numbers visible without scrolling (NFR-17, NFR-18). |
| Colour + icon for state indication          | Ensures accessibility for colour-blind users and aligns with WCAG guidance.        |
| Confirmation dialog for all control actions | Prevents accidental commands in a safety-sensitive environment (FR-45, FR-48, FR-50). |
| Hiding inaccessible menu items              | Reduces cognitive load for non-admin users and prevents confusion about locked features. |
| UTC stored, local time displayed            | Ensures data integrity across time zones while presenting operationally meaningful timestamps (FR-4). |
| Mid-fidelity wireframes                     | Balances speed of stakeholder review with sufficient detail for developer and QA alignment. |
| Progressive disclosure (portfolio → site → device) | Matches the mental model of facility managers moving from summary to detail as needed. |
